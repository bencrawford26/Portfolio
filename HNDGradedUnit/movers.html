Walkers

One of the main focuses of The Nature Of Code book is to familiarize its readers with simulating physics engines. At the early stages of the book this was reinforced by creating several different simulations of walkers (an individual pixel that is redrawn randomly around the screen to give the impression of movement) and movers (a circle which is drawn on screen that responds to things in the environmet or has a set movement path). Each application gets slightly more complicated or robust as the exercises move forward. I have created this page to store all of these apps and explain the features and differences between each one.

Here is the first example of a Random Walker I created for this project. I used this first as it is the "classic" example of a random walker, a random number between 0 and 4 is generated for each frame of animation. A simple series of coniditional statements is then used to determine which direction the walker should move in (up, down, left or right).

This walker was created to show my understanding of probability. Just as before the walker has a chance to move in any direction but the condtional statements dictate that the walker has a higher chance to move down or right rather than anywhere else. (hence why the walker in this example almost always moves its way off to the bottom right of the screen)

Finally, this example of a random walker uses Perlin Noise to calculate it's direction and the size of each "step" it takes. This was the first example of noise that I had used and as you can see it produces much smoother random values than the basic random function used in the "classic" walker. More details and examples of this can be found on the Perlin Noise page.

Movers

This example shows a mover object with it's location being governed by Perlin Noise. This is very similar to the previous walker example shown above but the background in this example is wiped each time the mover is drawn which removes the movers trail.

Here we have a mover object which is constantly accelerating towards the mouse. Wherever you position your mouse within that canvas the mover will set its course for! The mover never stops because it will always overshoot the mouses' location and have to turn back.

Last of all, this time there is an array of mover objects being pushed around the screen by two seperate "wind forces" (one from the top left of the screen and one from the bottom right). I used an array in the example to help visualize the wind current being simulated. Also, the mover objects vary in size and mass so some are affected by the wind force more than others. You'll notice this much more after the simulation runs for a while and the movers become wildly dispersed and lose their pattern.
